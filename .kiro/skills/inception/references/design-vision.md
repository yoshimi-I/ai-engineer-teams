# デザインビジョン（条件付き）

プロダクトが UI を持つ場合、アーキテクチャ設計に入る前にデザイン方針を
ユーザーとすり合わせる。技術スタック（UIキット、CSS戦略、コンポーネント
ライブラリ）の決定は、ここで合意したトーンに従う。

## 実行条件

- フロントエンド（Web/モバイル/デスクトップ）がスコープに含まれる
- ユーザー向けの画面・ダッシュボード・管理UI を作る
- メール/通知/印刷物など、ユーザーの目に触れるビジュアル要素がある

## スキップ条件

- 純粋な API / バックエンドサービス
- CLI / バッチ / ETL のみ
- SDK / ライブラリの公開
- 既に確立したデザインシステムに組み込む（そのシステム名を記録して終了）

## ステップ

### 1. コンテキストを確認
要件と（生成済みなら）ユーザーストーリーを読む。ターゲットユーザー、
ユースケース、ブランドの有無を把握してから質問する。

### 2. チャットでデザイン方針を質問する

関連する質問をグループ化し、1メッセージにつき最大 3〜5 問。選択肢形式
を基本に、該当なしの場合は自由記述も受け付ける。

**質問例**:

```
デザインの方向性について教えてください：

1. プロダクトのトーンに近いのはどれですか？
   A) プロフェッショナル / 落ち着いた（Linear、Stripe、Notion）
   B) フレンドリー / 親しみやすい（Airbnb、Slack）
   C) ミニマル / 静か（Apple、Vercel）
   D) 表現豊か / 個性的（Figma、Raycast）
   E) 業務・データ密度重視（Bloomberg、Datadog、dashboard系）

2. 参考にしたい既存プロダクトはありますか？（複数可）
   （例: "Linearの速度感 + Notionの余白" のような合成もOK）

3. カラー戦略は？
   A) 単色アクセント + ニュートラル中心
   B) ブランドカラー強め
   C) モノクロ / ほぼ白黒
   D) ダークモード前提

4. 情報密度は？
   A) データ密度高め（ダッシュボード・管理画面）
   B) バランス型（一般的なSaaS）
   C) 余白重視（マーケ、コンテンツ系）
   D) フォーカス型（エディタ、1タスク1画面）

5. デバイス優先度は？
   A) デスクトップ優先（作業ツール）
   B) モバイル優先（コンシューマー）
   C) 両方同等（レスポンシブ必須）
```

必要に応じて深掘り:

```
6. タイポグラフィの好みは？
   A) システムフォント（ネイティブ感）
   B) サンセリフ（Inter / Geist 等）
   C) セリフ（読み物寄り）
   D) モノスペース混在（エンジニア向け）

7. モーション/アニメーションは？
   A) 静的（装飾的アニメなし）
   B) 控えめ（fade / ease のみ）
   C) 表現豊か（spring、stagger、演出あり）

8. アクセシビリティ基準は？
   A) WCAG 2.1 AA（推奨ベースライン）
   B) WCAG 2.1 AAA（医療・公共・アクセシビリティ重視）
   C) 基本対応のみ（コントラスト + キーボード + alt）

9. UIキットの前提は？
   A) shadcn/ui + Tailwind（カスタム寄り）
   B) Material UI / Chakra / Mantine（既製コンポーネント）
   C) ヘッドレス（Radix / Ark UI）＋ 独自スタイル
   D) 完全カスタム
   E) お任せ（要件に合わせて提案してほしい）

10. ブランドアセットはありますか？
    - ロゴ、カラーパレット、タイポ指定、ブランドガイドライン等
    - ある場合: 配置パスまたはリンクを教えてください
    - ない場合: INCEPTION内で最小限のトーンだけ決めます
```

### 3. 合意内容を要約して承認を得る

チャットで以下を短くまとめ、ユーザーの承認を待つ:

```
デザインビジョン（ドラフト）

- トーン: プロフェッショナル / 落ち着いた
- 参考プロダクト: Linear, Stripe
- カラー: 単色アクセント（#5B5BFF） + ニュートラル
- 情報密度: バランス型
- デバイス: デスクトップ優先、モバイル対応必須
- タイポ: Inter（システムフォント fallback）
- モーション: 控えめ
- アクセシビリティ: WCAG 2.1 AA
- UIキット: shadcn/ui + Tailwind
- ブランドアセット: なし（INCEPTION で決めた最小セット）

これで進めてよいですか？
```

### 4. ドキュメントを生成

承認後、`aidlc-docs/inception/design/design-vision.md` に記録:

- **トーン**: 目指す雰囲気と、避けたい雰囲気
- **参考プロダクト**: 名前と、「どの要素を参考にするか」
- **デザイントークン（初期値）**: カラー、タイポ、スペーシング、角丸、シャドウ
- **密度・レイアウト方針**: コンテナ幅、グリッド、余白スケール
- **モーション原則**: ease、duration、使用シーン
- **アクセシビリティ基準**: 目標レベルと具体的なチェック項目
- **UIキット選定**: 採用したキットと理由、代替案を却下した理由

### 5. アーキテクチャ設計に合意内容を引き継ぐ

`architecture-design.md` の技術スタック選定では、ここで決めた方針を
優先制約とする。例:

- UIキット = shadcn/ui → React + Tailwind が必須
- モーション = 表現豊か → framer-motion を採用候補
- デバイス = モバイル優先 → PWA / Capacitor を検討

### 6. 監査に記録

`aidlc-docs/audit.md` にタイムスタンプ付きで追記。

## スキルへの引き継ぎ

INCEPTION 完了後、実装フェーズでは以下のスキルが
`aidlc-docs/inception/design/design-vision.md` を参照する:

- `frontend-design` — 具体的な UI 設計指針
- `baseline-ui` — Tailwind 制約
- `fixing-accessibility` — A11y チェック
- `fixing-motion-performance` — アニメーション性能
