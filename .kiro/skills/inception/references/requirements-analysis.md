# 要件分析（適応型）

常に実行。深度は複雑さに応じて適応。

## 適応的深度

| レベル | 条件 | 内容 |
|-------|------|------|
| 最小 | シンプルで明確なリクエスト | 意図分析のみ |
| 標準 | 通常の複雑さ | 機能要件 + 非機能要件 |
| 包括的 | 複雑、高リスク | トレーサビリティ付きの完全な要件 |

## ステップ

### 1. ユーザーリクエストを分析（意図分析）
- ユーザーは何を作ろうとしているか？
- どんな問題を解決するか？
- ターゲットユーザーは誰か？
- 成功基準は何か？

### 2. 深度を決定
基準: 機能数、統合、ユーザータイプ、リスクレベル。

### 3. チャットで明確化の質問をする
会話で直接ユーザーに質問。選択式フォーマットを使用。
**以下のカテゴリから、プロジェクトに関連するものを選んで質問する。**
1回のメッセージで3-5問まで。回答を待ってから次のグループへ。

#### 質問カテゴリ（該当するものを選択して質問）

**A. 認証・ユーザー管理**（ユーザーログインがある場合は必須）
- 認証方式: メール+パスワード / ソーシャルログイン(Google,GitHub等) / SSO / マジックリンク
- 認証基盤: 自前実装 / Cognito / Firebase Auth / Auth0 / Supabase Auth
- MFA（二要素認証）: 必要 / 不要 / 将来対応
- ユーザーロール: 単一 / 管理者+一般 / カスタムロール

**B. デプロイ・インフラ**（必須）
- クラウド: AWS / GCP / Vercel / Cloudflare / セルフホスト
- リージョン: 東京 / US / マルチリージョン
- 環境数: dev+prod / dev+staging+prod
- 予算感: 無料枠内 / 月$50以下 / 月$200以下 / 制限なし
- カスタムドメイン: あり（ドメイン名） / なし（デフォルトURL）

**C. 外部サービス連携**（該当する場合）
- 決済: Stripe / PayPal / なし
- メール送信: SES / SendGrid / Resend / なし
- ファイルストレージ: S3 / Cloudflare R2 / なし
- AI/ML: OpenAI / Anthropic / Bedrock / なし
- 検索: Algolia / Elasticsearch / DB全文検索 / なし
- リアルタイム: WebSocket / SSE / Pusher / なし

**D. データ**（DBを使う場合は必須）
- 初期データ/seed: 必要（何のデータ？） / 不要
- 既存データの移行: あり（移行元は？） / なし（新規）
- バックアップ: 自動日次 / 手動 / 不要（開発段階）

**E. 多言語・国際化**（該当する場合）
- 対応言語: 日本語のみ / 日英 / 多言語
- 翻訳方式: ハードコード / i18nライブラリ / CMS
- 日付/通貨フォーマット: ロケール対応必要か

**F. パフォーマンス・スケール**（本番運用する場合）
- 想定ユーザー数: ~100 / ~1000 / ~10000 / それ以上
- 同時接続: ~10 / ~100 / ~1000
- レスポンス目標: 特になし / 200ms以下 / 50ms以下
- SSR/SSG: 必要（SEO重要） / 不要（SPA）

**G. モニタリング・運用**（本番運用する場合）
- エラー追跡: Sentry / Bugsnag / CloudWatch / なし（開発段階）
- ログ: CloudWatch / Datadog / stdout のみ
- アラート: Slack通知 / メール / なし

**H. テスト戦略**
- E2Eブラウザ: Playwright / Cypress / なし
- CI時間の許容: 5分以内 / 10分以内 / 制限なし
- テストカバレッジ目標: 特になし / 80%以上 / 重要パスのみ

**I. 法的・コンプライアンス**（該当する場合）
- 個人情報: 取り扱う（プライバシーポリシー必要） / 取り扱わない
- Cookie同意: 必要（EU向け） / 不要
- 利用規約: 必要 / 不要

**J. 既存資産**（該当する場合）
- 既存コード: あり（リポジトリURL） / なし（ゼロから）
- 既存DB: あり（スキーマ提供可能？） / なし
- デザインアセット: Figma / 画像ファイル / なし

#### 質問の進め方

1. まずプロジェクトの概要を聞いた後、関連するカテゴリを判断
2. **必須カテゴリ**（B: デプロイ）は必ず質問
3. ユーザーログインがあれば A も必須
4. DBがあれば D も必須
5. 残りは該当する場合のみ
6. 「特に希望がなければこちらで決めます」の選択肢も常に提示

### 4. 要件ドキュメントを生成
`aidlc-docs/inception/requirements/requirements.md` を作成:
- 機能要件（機能領域ごとにグループ化）
- 非機能要件（パフォーマンス、セキュリティ、スケーラビリティ）
- 制約と前提
- スコープ外

### 5. サマリーを提示して承認を得る
チャットで簡潔なサマリーを表示。次に進む前にユーザーの確認を求める。

### 6. 監査に記録
`aidlc-docs/audit.md` にタイムスタンプ付きで追記。
